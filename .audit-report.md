| 项目 | 文件:行 | 说明 |
|------|---------|------|
| slug() 防路径穿越 | `store.go:258-259` | 只保留 `[a-z0-9-]`，`slug("../../../etc")` → `"etc"` |
| WriteDoc 白名单 | `memory.go:92-103` | `allowedDocPaths()` 限定只写已知记忆文件 |

### 🔴 风险发现

#### P2 — Store.reindex/flushIndex 写入竞争
**文件**: `store.go:165-190`
**问题**: `indexLinesExcept()` 读 MEMORY.md 后，`flushIndex()` 写回之间无锁保护。并发 Save/Delete 会丢失索引行。
```go
func (s Store) indexLinesExcept(name string) map[string]string {
    existing, _ := os.ReadFile(filepath.Join(s.Dir, indexFile))  // read
    // …没有锁…
}
func (s Store) flushIndex(lines map[string]string) error {
    return os.WriteFile(filepath.Join(s.Dir, indexFile), …)      // write
}
```
**修复**: 给 Store 加 `sync.Mutex` 包裹 reindex + flushIndex 序列。

#### P3 — @import 可读任意文件
**文件**: `doc.go:202-227`
**问题**: `resolveImports` 对 `@<path>` 行直接用 `os.ReadFile(path)` 读取。一个恶意的记忆文件（或用户意外包含）可读取 `.env`、SSH 密钥等敏感文件注入 prompt。
```go
b, err := os.ReadFile(path)  // 无路径限制，可读任意用户可读文件
```
**缓解**: 这是设计特性（类似 shell source），但用户应知晓风险。建议在文档中明确警告。

#### P3 — 记忆正文直接注入系统 prompt 无净化
**文件**: `memory.go:136-146`
**问题**: `Block()` 将记忆文件正文直接渲染进系统前缀。恶意的记忆文件可注入 prompt 覆盖指令。
**缓解**: 记忆文件是用户自主编写的，风险由信任模型控制。

### ✅ 安全路径
| 项目 | 文件:行 | 说明 |

| 世界可写目录拒绝 | `manager.go:168-176` | 拒绝从 /tmp 等目录启动 LSP 服务器 |

### 🔴 风险发现

#### P2 — LSP 服务器输出未经净化即返回模型
**文件**: `results.go:110-112`, `manager.go:297-307`
**问题**: LSP 诊断信息、hover 文本直接格式化到工具结果。恶意/被攻陷的 LSP 服务器可通过构造的诊断消息注入 prompt 指令。
```go
fmt.Fprintf(&b, "%d:%d %s%s %s\n",
    d.Range.Start.Line+1, d.Range.Start.Character+1, sev, src,
    strings.TrimSpace(d.Message))  // 不可信的 LSP 服务器输出
```
**修复**: 对 LSP 返回的文本应用 `sanitizeHookOutput` 类似的净化（控制字符过滤 + 长度截断）。

#### P3 — abs() 对超工作区路径静默重写
**文件**: `manager.go:117-119`
**问题**: 绝对路径超出 `wsRoot` 时，不返回错误而重写为工作区内路径（大概率不存在），导致后续错误信息模糊。
```go
rel, err := filepath.Rel(m.wsRoot, cleaned)
if err != nil || strings.HasPrefix(rel, "..") {
    return filepath.Join(m.wsRoot, cleaned)  // 静默重写
}
```

#### P3 — LSP 二进制仅凭 PATH 解析，无完整性校验
**文件**: `manager.go:164`
**问题**: `exec.LookPath(spec.Command)` 取 PATH 上第一个匹配。`isWorldWritable` 只检查二进制所在目录，不覆盖 PATH 中所有目录。
```go
bin, err := exec.LookPath(spec.Command)
```

### 🔴 风险发现

#### P0 — 信任的项目钩子 = 任意 shell 执行
**文件**: `hook.go:465-469`
**问题**: `DefaultSpawner` 将命令包装为 `sh -c <command>`（Windows 为 `cmd /c`）。一旦用户信任一个项目，其 `.reasonix/settings.json` 中的钩子拥有用户完整权限的 shell 访问。
```go
func shellInvocation(command string) (string, []string) {
    return "sh", []string{"-c", command}  // 完全 shell 执行
}
```
**缓解**: 信任门控（trust.go）设计良好：信任标志存储在用户全局 `~/.reasonix/trust.json`，按绝对路径索引。前端应在信任前显示钩子列表。**这是整个系统风险最高的攻击面。**

#### P1 — PreToolUse 钩子可任意修改工具参数
**文件**: `hook.go:393-399`
**问题**: PreToolUse 钩子 exit 0 且 stdout 输出合法 JSON 时，该 JSON **替换后续钩子和实际调用的工具参数**。恶意钩子可将 bash 命令替换为任意代码。
```go
if s := strings.TrimSpace(r.Stdout); s != "" && json.Valid([]byte(s)) {
    payload.ToolArgs = json.RawMessage(s)    // 参数修改
    report.ModifiedArgs = json.RawMessage(s)
}
```

#### P2 — 配置损坏静默吞没
**文件**: `hook.go:202-212`
**问题**: `readSettings` 对任何 JSON 解析错误返回 nil，`Load` 静默跳过。用户不慎损坏 settings.json 会丢失所有钩子且无提示。
```go
if err := json.Unmarshal(b, &s); err != nil {
    return nil  // 静默吞没
}
```

#### P2 — trust.json 无完整性校验
**文件**: `trust.go:67-77`
**问题**: `trust.json` 以明文 JSON 存储（0o600）。同一用户的任何进程可修改它，将任意项目标记为已信任。无 HMAC 或签名保护。
```go
return os.WriteFile(path, b, 0o600)  // 无完整性保护
```

### 🔴 风险发现

#### P1 — Cron 任务文件路径来自环境变量，无校验
**文件**: `cron.go:31-36`
**问题**: `REASONIX_CRON_TASKS_PATH` 环境变量直接用作文件路径，无路径遍历检查、无路径净化。攻击者控制此环境变量可读写任意 JSON 文件。同时 `REASONIX_CHAT_ID`（cron.go:39）控制新任务的 chat_id 归属。
```go
func getCronTasksPath() (string, error) {
    path := os.Getenv("REASONIX_CRON_TASKS_PATH")
    // 无路径校验，无遍历防护
    return path, nil
}
```
**修复**: 对路径执行 `filepath.Clean()` + 限制在已知目录；验证 `REASONIX_CHAT_ID` 为合法用户 ID。

#### P2 — Cron 任务 JSON 文件无完整性保护
**文件**: `cron.go:96-103`
**问题**: Cron 任务以纯 JSON 存储，无签名/HMAC。有文件系统访问权限的用户可直接编辑注入任意定时任务，定时执行恶意 prompt。
```go
var tasks []*CronTask
if err := json.Unmarshal(data, &tasks); err != nil { … }
// 无完整性校验
```

#### P2 — 后台作业无权限隔离
**文件**: `jobs.go` 全局
**问题**: 所有 bash/task 后台作业以 reasonix 进程相同的 OS 用户身份和权限运行。无沙箱/容器化。一个被攻陷的作业可访问用户所有文件。

#### P2 — Cron 任务 prompt 无大小限制
**文件**: `cron.go:170`
**问题**: `scheduleTask` 接受无大小上限的 prompt 参数。超大 prompt 可被存储并重复执行，导致资源耗尽。
```go
Prompt string `json:"prompt"`  // 无大小校验
```

#### P3 — 作业 ID 可预测
**文件**: `jobs.go:209`
**问题**: 作业 ID 格式为 `fmt.Sprintf("%s-%d", kind, m.seq)`，seq 为递增计数器。
```go

id := fmt.Sprintf("%s-%d", kind, m.seq)
```

# Permission / YOLO evidence

## Policy.Decide precedence (permission.go:131-145)
```
case matchAny(p.Deny, ...): return Deny
case matchAny(p.Ask, ...):  return Ask
case matchAny(p.Allow, ...):return Allow

```
deny > ask > allow > fallback. Correct precedence.

## Gate.Check (permission.go:258-298)
- bash non-readonly: if isReadOnlyBashSubject(subject) → readOnly=true (line 259-263)
- Deny → returns false (line 266)
- Ask with nil Approver → returns true (line 269-271) "non-interactive: preserve autonomy"
- Ask with Approver → calls Approve (line 273)

## YOLO bypass (controller.go:139-144, 1862-1872, 1913-1926)
- bypass bool, session-scoped, never persisted. Set via --dangerously-skip-permissions / --yolo (cli.go:397-398)
- gateApprover.Approve: if c.bypass → return true,false,nil (line 1866-1870)
- requestApproval: if c.bypass || c.granted[key] → return true (line 1923, 1935)
- Deny rules "already bit before this point" (comment line 1864) — confirmed: Gate.Check evaluates Deny BEFORE calling Approver.

## Ask tool NOT bypassed in YOLO (controller.go:714-715, yolo_test.go:114-200)
- Ask tool genuine user decision; YOLO only auto-approves tool calls.

## Prompt-injection resistance
- Policy is pure rule eval on (toolName, subject from JSON args). subjectKeys: command/file_path/path/pattern (permission.go:177).
- The model cannot change its own toolName or restructure args to bypass — args come as raw JSON to Gate.Check. BUT: for bash, subject=command string; a malicious command could be crafted to match an allow glob. E.g. allow rule "bash(go build*)" — model could run "go build; curl evil". containsShellSyntax (bash_readonly.go:98-102) only forces NON-readonly for shell syntax (;|&<>\n`$()${), but allow rules are glob-matched against the FULL subject string regardless of shell syntax. So "go build; curl evil" matches glob "go build*" and is ALLOWED without prompt (if such allow rule exists). This is a real injection vector when users add broad allow globs.

## note #2 · 2026-06-19T03:20:02.29189542Z · kind=evidence

# MCP plugin security evidence

## stdio transport (transport_stdio.go)
