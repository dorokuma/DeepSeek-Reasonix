# 白名单/权限控制分布审计

## 1. 总体架构：三层权限体系

```
┌─────────────────────────────────────────────────┐
│  ① MainAgentAllowed (config → map[string]bool)  │  ← 工具可见性（主代理看到哪些工具）
│     仅根 agent（depth=0），nil = 全部可见         │
├─────────────────────────────────────────────────┤
│  ② Policy (permission.Policy → Gate.Check)      │  ← 每次调用运行时判定
│     Allow/Ask/Deny 规则 + 回退模式                │
├─────────────────────────────────────────────────┤
│  ③ AllowedTools (skill.Skill.AllowedTools)      │  ← 子代理技能的工具集裁剪
│     FilterRegistry 从父 registry 中选取子集       │
└─────────────────────────────────────────────────┘
```

---

## ② Policy（内部 permission 模块）

### 定义位置
- **`internal/permission/permission.go`**
  - `type Policy struct { Mode Decision; Allow, Ask, Deny []Rule }`
  - `func New(mode string, allow, ask, deny []string) Policy`

### 配置加载
- **`internal/config/config.go:618-629`** → `PermissionsConfig`
  ```toml
  [permissions]
  mode = "ask"          # 回退策略：allow / ask / deny
  allow = []            # 工具白名单：如 "bash", "bash(cargo build)"
  ask = []              # 需审批的工具
  deny = []             # 禁止的工具
  main_agent_allowed = []  # 根代理工具可见性（非 nil 时启用）
  ```
- **`internal/boot/boot.go:407`** → `policy := permission.New(cfg.Permissions.Mode, cfg.Permissions.Allow, cfg.Permissions.Ask, cfg.Permissions.Deny)`

### Policy 能控制什么
- **工具级别**：`"bash"` 匹配所有 bash 调用
- **工具+主题（glob）**：`"bash(cargo build)"`, `"write_file(/etc/*)"`, `"bash(rm*)"`
- **分类别名**：`"Edit"` 展开为 `{"move_file", "write_file"}`
- **只读/写入自动区分**：只读工具永远 fallback 到 Allow；写入工具 fallback 到 Mode
- **审批级别**：Allow（自动放行） / Ask（弹窗） / Deny（硬阻止）

### Policy 的粒度限制
- 不支持通配符工具名称（如 `"tool_*"`）
- 不支持跨工具权限（不能在一个规则里同时控制 `bash` 和 `write_file`）
- `main_agent_allowed` 是**粗粒度工具可见性**（仅 root agent），不是运行时权限

### 运行时消费流程
1. `agent.go:1207-1222` → 每次工具调用执行 `a.gate.Check(ctx, call.Name, args, readOnly)`
2. `gate.Check` → `policy.Decide(toolName, readOnly, args)` 返回 Allow/Ask/Deny
3. `controller.go:736` → 交互模式下 `EnableInteractiveApproval()` 用带 Approver 的 Gate 替换

### 区别于 Policy
- MainAgentAllowed 控制**工具对模型可见性**（schema 不暴露），不是运行时权限
- 只作用于根代理（depth=0），子代理不受影响
- 设为空/nil 时无限制

## 2. AllowedTools vs Policy：关系与冗余分析

### 关系
```
AllowedTools (技能定义)
     ↓
FilterRegistry (构建子 registry)
     ↓
子代理拥有缩减后的工具集合
     ↓
子代理每次调用 → Gate.Check (同一份 Policy)
```

- **AllowedTools 是编译时（注册时）裁剪**：决定子代理能看到/调用哪些工具
- **Policy 是运行时拦截**：无论工具是否在 AllowedTools 中，每个调用都要过 Gate.Check
- **两者作用正交**，但功能上有重叠

### AllowedTools 是否多余？

**不完全多余**，原因有二：

1. **安全性**：AllowedTools 是**白名单模型**——只列出允许的工具。如果一个新工具被注册但忘了加入 Policy deny 规则，AllowedTools 仍然阻止它。Policy 的默认回退是 `ask`（写入工具），但只读工具默认 Allow。

2. **UI/模型体验**：工具 schema 不传给子代理，模型不会看到/尝试未授权的工具。Policy 拒绝会触发模型重试，浪费 token。

3. **语义清晰**：review 技能只应该有读 + bash 工具，不让它看到 `task` / `run_skill` 等元工具。这已经通过 `SubagentMetaTools()` 做了，但 AllowedTools 提供了更精细的裁剪。

### 但如果 Policy 足够完善，AllowedTools 可移除的场景

如果 Policy 配置为：
```toml
[permissions]
mode = "ask"  # 写入工具弹窗
ask = []
deny = ["task", "run_skill", "install_skill", "install_source", "explore", "research", "review", "security_review"]
```
那 AllowedTools 的裁剪就冗余了——子代理虽然能看到 `task` 等工具，但调用会被 Policy 拒绝。

但**只读工具默认 Allow**，所以如果不加 AllowedTools，子代理能看到父级的所有只读工具（包括可能不应该暴露的）。这意味着要移除 AllowedTools，必须配置 `deny` 列表覆盖所有不希望子代理用的工具——而这需要用户手动维护。

## 3. FilterRegistry 调用汇总

| 位置 | 传入的 names | exclude | 说明 |
|---|---|---|---|
| `boot.go:527` | `sk.AllowedTools` | `SubagentMetaTools()` | 子代理技能 runner |
| `task.go:264` | `names` (来自 task 参数或空) | 无 (allowMeta=true) | task 工具可选 allowMeta |
| `task.go:266` | `names` (来自 task 参数或空) | `SubagentMetaTools()` | task 工具默认行为 |
| `task.go:307` | 所有父工具名 - `plannerNonResearchTools` | `SubagentMetaTools()` + `plannerNonResearchTools` | 计划模式 |

### 能否去掉 AllowedTools，让所有子代理统一用 Policy？

**理论可行但需要改动：**

1. **去掉 `skill.Skill.AllowedTools` 字段** 和 frontmatter 解析
2. **去掉 `FilterRegistry` 的 names 参数**，改用 `names = parent.Names()`（所有工具）
3. **配置适当的 deny 规则** 阻止子代理不应调用的工具
4. **移除 `subagentMetaTools` 硬编码**（或移到默认 Policy deny 列表）
5. **更新 `cli/review.go`** 不再用 AllowedTools 构建 registry

**直接影响**：
- 子代理模型会看到全部工具 schema（更多 token 消耗）
- 需要用户维护更完整的 Policy deny 列表
- 失去"默认禁止，显示允许"的安全模型

**建议**：不直接移除 AllowedTools，而是：
- 将内置技能的 AllowedTools 与 Policy deny 列表对齐
- 确保 Policy 的 deny 规则覆盖 `subagentMetaTools`
- 让 AllowedTools 作为**可选的性能优化**（减少 schema 大小），而不是安全机制

## 5. 结论：统一建议

### 现状
- 三套独立机制，职责部分重叠
- `AllowedTools` + `SubagentMetaTools()` 构建"安全子集"，Policy 做"最终闸门"

### Policy 能力边界
| 能力 | 支持 |
|---|---|
| 工具名精确匹配 | ✅ |
| 工具+主题 glob | ✅ `bash(rm -rf *)` |
| 工具分类别名 | ✅（`"Edit"` → `{move_file, write_file}`） |
| 通配符工具名 | ❌ 不支持 `"tool_*"` |
| 只读/写入不同回退 | ✅ |
| 运行时动态添加规则 | ✅（remember 机制） |
| 用户记忆持久化 | ✅ |

### 推荐方案
**不移除 AllowedTools**，而是：
1. **标准化内置技能的 AllowedTools**：确保 `extraReadTools` 机制清晰，内置工具列表与 Policy 默认 deny 列表对齐
2. **加强 Policy 默认 deny**：在默认配置中将 `subagentMetaTools` 加入 deny 列表
3. **文档化三层关系**：让用户理解 MainAgentAllowed / Policy / AllowedTools 的层次
4. **未来可选**：将 AllowedTools 改为 Policy 的编译期推导，而非独立配置
