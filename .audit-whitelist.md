# Investigation: Why `glob` was callable despite whitelist restrictions

## 1. `DefaultMainAgentAllowed` includes `glob`

File: `/root/reasonix/internal/agent/agent.go`, lines 39-70

```go
var DefaultMainAgentAllowed = map[string]bool{
    "task":             true,
    ...
    "glob":             true,    // line 60-61
    ...
}
```

If using the DEFAULT whitelist (no custom config), `glob` IS allowed. The user would only see a problem if they configured a custom `main_agent_allowed` list that omits `glob`.

## 2. Boot initialization: custom vs default

File: `/root/reasonix/internal/boot/boot.go`, lines 452-460

```go
var mainAgentAllowed map[string]bool
if len(cfg.Permissions.MainAgentAllowed) > 0 {
    mainAgentAllowed = make(map[string]bool)
    for _, name := range cfg.Permissions.MainAgentAllowed {
        mainAgentAllowed[name] = true
    }
} else {
    mainAgentAllowed = agent.DefaultMainAgentAllowed
}
```

When a custom `main_agent_allowed` list is provided in the config, ONLY those tools are in the whitelist — the defaults are NOT merged. If `glob` is absent from the custom list, the main agent should be blocked from calling it.

## 3. The two whitelist check points (both in `agent.go`)

### 3a. Schema filtering — `getSchemasForContext` (line 943-955)

```go
func (a *Agent) getSchemasForContext(ctx context.Context) []provider.ToolSchema {
    schemas := a.tools.Schemas()
    if allow := a.mainAgentAllowed; allow != nil && NestingDepthFrom(ctx) == 0 {
        filtered := make([]provider.ToolSchema, 0, len(schemas))
        for _, s := range schemas {
            if allow[s.Name] {
                filtered = append(filtered, s)
            }
        }
        return filtered
    }
    return schemas
}
```

**At depth 0 (main agent):** Only whitelisted tool schemas are shown to the model → model doesn't even know `glob` exists.

**At depth > 0 (sub-agent):** The `NestingDepthFrom(ctx) == 0` condition is FALSE → ALL schemas are returned regardless of whitelist.

### 3b. Execution guard — `executeOne` (line 1192-1200)

```go
// Main-agent whitelist: when nesting depth is 0 (root/main agent),
// only allow explicitly permitted tools (if the option is set).
if allow := a.mainAgentAllowed; allow != nil && NestingDepthFrom(ctx) == 0 && !allow[call.Name] {
    return toolOutcome{
        output:  fmt.Sprintf("permission denied: tool %q not allowed for main agent", call.Name),
        blocked: true,
        errMsg:  fmt.Sprintf("permission denied: tool %q not allowed for main agent", call.Name),
    }
}
```

Three conditions must ALL be true to block:
1. `a.mainAgentAllowed != nil` — whitelist is configured
2. `NestingDepthFrom(ctx) == 0` — root agent (depth 0)
3. `!allow[call.Name]` — tool NOT in the whitelist

**When ANY condition is false, the check is skipped entirely.**

## 4. Sub-agents bypass the whitelist

### 4a. Sub-agent creation in `task.go`

File: `/root/reasonix/internal/agent/task.go`, lines 230-237 (background) and 246-250 (foreground):

```go

// Foreground task (line 246):

```

Both increment nesting depth by 1, so sub-agents always have `NestingDepthFrom(ctx) >= 1`.

Since `a.mainAgentAllowed` is nil, the condition `allow != nil` at line 1194 is false → whitelist check is completely bypassed.

### 4c. `NestingDepthFrom` default

File: `/root/reasonix/internal/agent/context.go`, lines 19-28:

```go

        if d, ok := v.(int); ok {
            return d
        }
    }
    return 0  // default when not set — matches the main agent

```

## 5. How sub-agents call `glob`

The `task` tool's sub-registry (line 257-262 in `task.go`) inherits all parent tools EXCEPT meta-tools:

```go
var subagentMetaTools = []string{
    "task", "run_skill", "read_skill", "install_skill",
    "install_source", "explore", "research", "review", "security_review",
}
```

`glob` is NOT in this exclusion list, so it IS available to sub-agents.

## Root Cause Summary

There are **two independent bypass mechanisms** that allow a sub-agent (spawned via the `task` tool) to call `glob` even when the main agent's whitelist excludes it:

1. **`NestingDepthFrom(ctx) != 0` skips the check**: Both `getSchemasForContext` and `executeOne` guard the whitelist with `NestingDepthFrom(ctx) == 0`. Sub-agents always have depth >= 1, so the whitelist filter is skipped at both the schema-presentation and execution-enforcement layers.

2. **`MainAgentAllowed` is not propagated to sub-agents**: `runSub` (and `RunSubAgent`) construct sub-agent Options without setting `MainAgentAllowed`. This means `a.mainAgentAllowed` is nil for sub-agents, which causes the `allow != nil` condition at line 1194 to be false, bypassing the check entirely even if depth were somehow 0.

**If the user observed the main agent (depth 0, executor) calling `glob`:** Check the config — `glob` is in `DefaultMainAgentAllowed` (line 60-61). If using defaults, it IS whitelisted. Only a custom `main_agent_allowed` list that omits `glob` would block it.

**If the user observed a sub-agent (background `task`) calling `glob`:** This is expected behavior — sub-agents are intentionally unrestricted. The `task` tool description in the schema says "(all 53 tools by default)" confirming this design.
