# chunkRing 代码审计报告

## 总览
`chunkRing` 是 OpenAI 和 Anthropic 两个 provider 中完全相同的 ring buffer 实现（代码重复拷贝），用于解耦 HTTP SSE 读取 goroutine 与消费者之间的背压。容量 `cap=64`，策略为「丢旧留新」（drop-oldest）。

---

## 发现

### P0 — close() 在 buffer 非空且无消费者读取时死锁

**问题描述：** `close()` 在关闭 stopCh 后等待 `<-cr.done`，但 drain goroutine 的 `drainAll()` 在尝试发送缓冲区内剩余 chunk 到 `out` 时，如果 `out` 满且 ctx 未被取消，则永久阻塞。`drainAll()` 内部仅检查 `<-cr.ctx.Done()`，不检查 `stopCh`。这导致 `close()` 永远无法返回。

**代码位置：** 
- `openai.go:327-340` (`drainAll`) — 内层 select 只检查 `ctx.Done()`
- `openai.go:370-381` (`close`) — 等待 `<-cr.done`，但 drain 被阻塞
- anthropic 相同位置

**触发条件（生产环境）：**
1. 消费者因 ChunkError 提前退出读取循环（agent.go:887）
2. 此时 buffer 中仍有未被 drain 的 chunk（例如 streamWithReconnect 发送的第 2 个 ChunkError）
3. ctx 未被取消
4. `close()` 被 defer 调用 → 死锁

**复现：** 我编写的压力测试 `TestChunkRingCloseRace` 在 30 秒后 timeout，stack trace 显示 `close()` 在 `<-cr.done` 处阻塞，而 drain goroutine 在 `drainAll()` 的 `cr.out <- chunk` 处阻塞。

**修复建议：** `drainAll()` 内层 select 应增加 `<-cr.stopCh` 分支，或给 `close()` 增加超时：

```go
// 方案 A：在 drainAll 内层 select 增加 stopCh
select {
case cr.out <- chunk:
case <-cr.ctx.Done():
    return
case <-cr.stopCh:
    return
}
```

```go
// 方案 B：close() 加超时保护（备选方案）
func (cr *chunkRing) close() {
    close(cr.stopCh)
    select {
    case <-cr.done:
    case <-time.After(5 * time.Second):
        // timeout: drain goroutine stuck, force exit
    }
    close(cr.out)
}
```

**严重性：P0** — 一旦触发，goroutine 永久泄漏，`close()` 永不返回。

---

### P1 — drainAll 在 ctx 取消时不保证 buffer 排空

**问题描述：** `drain()` 在收到 `<-cr.ctx.Done()` 后调用 `drainAll()`，而 `drainAll()` 内部也会因 `ctx.Done()` 提前返回，导致 buffer 中剩余 chunk 静默丢弃。注释声称「drain goroutine always drains remaining chunks before exiting」，但 ctx 取消路径下该保证不成立。

**代码位置：** `openai.go:348-365`、anthropic 对应位置

**影响：** 当用户取消请求（Ctrl+C）时，buffer 中最多 63 个 chunk（文本/推理/工具调用）可能丢失。工具调用 chunk 丢失会导致工具结果丢失（输出损坏），ChunkDone 丢失则消费者侧 channel 正常关闭（因为 close() 会 close(out)），但中间文本 chunk 丢失导致输出不完整。

**修复建议：** ctx 取消路径下仍应尝试发送剩余 chunk，直到 buffer 排空或超时：

```go
case <-cr.ctx.Done():
    // 尽力排空，不依赖 ctx.Done() 退出内层循环
    cr.drainAllWithTimeout(100 * time.Millisecond)
    return
```

**严重性：P1** — 仅在用户取消时触发，但可能导致输出不完整。

**代码位置：** 
- `openai.go:255-381`
- `anthropic.go:490-620`

**修复建议：** 将 chunkRing 提取到公共包（如 `internal/provider/chunkring.go`），两个 provider 共享同一份代码。

**严重性：P1** — 不直接影响正确性，但严重降低可维护性。

### P2 — send() 在 stopCh 关闭后有短暂竞态窗口

**问题描述：** `send()` 在检查 `stopCh` 之后、获取 mutex 之前存在竞态窗口。如果 `close()` 在此窗口内关闭 `stopCh`，chunk 会被添加到 buffer，但 drain goroutine 已退出（或即将退出），导致该 chunk 永不被消费。`send()` 返回 `true`（调用者认为已接受），但 chunk 实际丢失。

**代码位置：** `openai.go:291-321`（send 函数）

```go
select {
case <-cr.stopCh:  // 检查 stopCh
    return false
default:
}                     // ← 竞态窗口：此时 close() 关闭 stopCh
cr.mu.Lock()          // ← 锁获取
cr.buf = append(cr.buf, chunk)  // chunk 入队但无法被消费
cr.mu.Unlock()
```

**影响：** 调用者基于 `true` 返回值认为 chunk 已成功入队，实际可能丢失。在 `streamWithReconnect` 中，ChunkError 可能丢失（不过此时上下文已经结束，影响有限）。

**修复建议：** 在 mutex 保护下重新检查 stopCh：

```go
cr.mu.Lock()
select {
case <-cr.stopCh:
    cr.mu.Unlock()
    return false
default:
}
if len(cr.buf) >= cr.cap {
    cr.buf = cr.buf[1:]
}
cr.buf = append(cr.buf, chunk)
cr.mu.Unlock()
```

**严重性：P2** — 仅在并发 close + send 时触发，且丢失的通常是冗余的 ChunkError。

---

### P2 — 容量 64 的合理性未经验证

**问题描述：** `cap=64` 的选择没有注释说明依据。在特定场景下（消费者处理工具调用时停顿 + 模型快速生成大量短 chunk），64 个槽位可能不足以吸收背压，导致文本 chunk 丢失，输出不完整。

**代码位置：** `openai.go:213`、`anthropic.go:163`

**影响：** 文本 chunk 丢失 → 输出句子断裂。工具调用 chunk 丢失（ChunkToolCall、ChunkToolCallStart）→ 工具调用失败或前端 UI 异常。

**修复建议：** 
1. 注释说明 64 的选择依据
2. 考虑增大为 256 或 1024（chunk 结构体很小，~80 bytes，64*80=5KB，256*80=20KB，均可接受）
3. 增加监控：如果 buffer 满导致丢 chunk，记录日志以便调优

**严重性：P2** — 需要实际数据来确认是否触发。

---

### P2 — slice 操作可能阻止 GC

**问题描述：** `cr.buf = cr.buf[1:]` 通过 re-slice 丢弃最旧 chunk。Go 的 slice 操作保留对底层数组的引用，被丢弃的 `buf[0]` 无法被 GC 回收，直到整个底层数组被回收。长期运行的流可能导致内存累积。

**代码位置：** `openai.go:309-312`、`anthropic.go:546-549`

**修复建议：** 在删除后显式清空引用：

```go
if len(cr.buf) >= cr.cap {
    cr.buf[0] = provider.Chunk{}  // allow GC
    cr.buf = cr.buf[1:]
}
```

**严重性：P2** — 实际内存泄漏量很小（每个 chunk ~80 bytes），累积最多 ~5KB。

---

### P2 — openai 的 streamWithReconnect 可能发送重复 ChunkError

**问题描述：** 当 `readStream` 返回非连接重置错误时，`streamWithReconnect` 发送 `ChunkError`。但某些错误路径下（如 `sawDone=false` 且 `emitted=true`），`readStream` 返回的 `StreamInterruptedError` 可能被重复包装并发送为两个 ChunkError（一个来自 `streamWithReconnect` 的非连接重置路径，但此路径实际仅发送一个）。

**代码位置：** `openai.go:234-236`

```go
if !provider.IsConnReset(err) {
    cr.send(ctx, provider.Chunk{Type: provider.ChunkError, Err: err})
    return
}
```

**影响：** 消费者收到两个 ChunkError（第一个来自 readStream 内部？实际上 openai 的 readStream 不发送 ChunkError，只有返回错误）。但 `streamWithReconnect` 在 `emitted=true` 路径下（行 238-240）发送的 `StreamInterruptedError` 被包装了两层，消费者侧可能混淆。

**修复建议：** 统一错误发送逻辑，避免 readStream 和 streamWithReconnect 双发。

**严重性：P2** — 消费者能正确处理多个 ChunkError。

---

### P3 — wakeCh 缓冲为 1 可能导致 drain 延迟

**问题描述：** `wakeCh` 容量为 1，当 drain goroutine 正在 drainAll 时，新的 wake 信号被丢弃。这不会导致问题（drainAll 会持续排空直到 buffer 空），但如果 drainAll 中途返回（ctx 取消），而新 chunk 在 drain 返回 select 之前到达，则 chunk 可能延迟被处理。

**代码位置：** `openai.go:279`、`anthropic.go:516`

**影响：** 在高吞吐场景下可能导致 chunk 短暂延迟，但不会丢失或乱序。

**严重性：P3** — 仅影响延迟，不影响正确性。

---

## 总体评价

`chunkRing` 的设计思路正确（解耦 HTTP 读取和消费者处理），基本功能（FIFO、丢旧留新、并发安全）实现良好。`-race` 测试无数据竞争。

**最严重的风险是 P0 死锁问题**，虽然生产环境触发条件较苛刻，但一旦触发会导致 goroutine 永久泄漏和 `close()` 永不返回。**强烈建议优先修复。**

次要问题包括 ctx 取消时 buffer 未排空、代码重复、无单元测试覆盖、以及容量选择缺少依据。
