## 1. `SplitMCPName` (tool.go:177–202)

### 1a. 不以 `mcp_` 开头时返回 ok=false
- **结果**: ✅ 通过
- **代码**: 第178-180行
  ```go
  if !strings.HasPrefix(name, MCPNamePrefix) {
      return "", "", false
  }
  ```
- 正确：当 name 不以 `"mcp_"` 开头时立即返回 `false`。

### 1b. `mcpPrefixes` 为 nil/空时安全处理（不会 panic）
- **结果**: ✅ 通过
- **代码**: 第181-185行
  ```go
  raw := mcpPrefixes.Load()
  prefixes, ok := raw.([]string)
  if !ok || len(prefixes) == 0 {
      return "", "", false
  }
  ```
- `atomic.Value` 在从未 Store 时 Load() 返回 `nil`。`nil.([]string)` 类型断言成功（nil 是有效的 nil `[]string`），`len(prefixes) == 0` 对 nil 也成立 => 不会 panic。

### 1c. 最长前缀匹配逻辑正确
- **结果**: ✅ 通过
- **代码**: 第187-195行
  ```go
  best := ""
  for _, p := range prefixes {
      if strings.HasPrefix(name, p) && len(p) > len(best) {
          best = p
      }
  }
  if best == "" {
      return "", "", false
  }
  ```
- 遍历所有前缀，用 `len(p) > len(best)` 比较保留最长匹配。当无匹配时返回 false。逻辑正确。

### 1d. server 名提取 `prefix[4:len(prefix)-1]`
- **结果**: ✅ 通过
- **代码**: 第200行
  ```go
  server = best[len(MCPNamePrefix) : len(best)-1]
  ```
- `MCPNamePrefix = "mcp_"` 长度为4。`len(best)-1` 去掉尾部的 `_`。例如 `"mcp_jina_"` → `best[4:9]` = `"jina"`，`"mcp_codegraph_"` → `best[4:14]` = `"codegraph"`。**假设所有前缀以 `_` 结尾**（由 `ToolPrefix` 保证，见下），正确。
- ⚠ **注意事项**：如果某个前缀不以 `_` 结尾，`len(best)-1` 会切掉最后一个字符而非尾部分隔符。但前缀的唯一来源是 `ToolPrefix()`（后文验证），它固定返回 `"mcp_" + normalizeName(server) + "_"`，因此安全。

### 1e. 并发安全性（`atomic.Value`）
- **结果**: ✅ 通过
- **代码**: 
  - 声明 `var mcpPrefixes atomic.Value` (第163行)
  - 写入: `mcpPrefixes.Store(prefixes)` (第170行)
  - 读取: `mcpPrefixes.Load()` (第181行)
- `atomic.Value` 保证 Store/Load 的 happens-before 语义。只有一个写入者（`RegisterMCPPrefixes`）+ 多个读取者（`SplitMCPName`），使用模式正确。

## 额外发现

### 问题 1: `Host.Remove` 的返回值与 `ToolPrefix` 重复逻辑
- **文件**: `plugin.go:806`
- **严重性**: ⚠️ 小建议
- **说明**: `Remove` 的 return 语句手写了 `"mcp_" + normalizeName(name) + "_"`，这等价于调用 `ToolPrefix(name)`。如果 `ToolPrefix` 的实现逻辑未来变化（例如添加了合法性校验），这里可能不同步。
- **建议**: 改为 `return ToolPrefix(name), true` 以消除重复。

### 问题 2: Auto-restart goroutine 忽略了 `Remove` 返回的 prefix（可能不是 bug，但需留意）
- **文件**: `plugin.go:745`
- **严重性**: ℹ️ 观察项
- **说明**: Auto-restart goroutine 内部调用 `h.Remove(s.Name)` 时未使用返回的 prefix 去清理工具注册表。随后 `h.Add(restartCtx, s)` 返回的 tools 也未注册到全局 `Registry`。但该 goroutine 在 `Add` 内部，而 `Add` 的调用者负责管理注册表，重启后需要重新 Add tools 到 reg。
- **建议**: 确认 auto-restart 场景下工具注册表的清理和重建是否由调用链路保证。
