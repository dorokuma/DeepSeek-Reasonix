# Bridge 实现审计报告

依据 `/root/docs/spark/2026-07-12-bridge-redesign.md` §十三 验收清单（19 条）逐条核查。

---

## 1. 对话 draft → 全文正式消息（过长拆段），无悬空 draft
**状态：✅ 已实现**

- **拆段逻辑**：`splitTelegramText()`（telegram_text.go:59-99）按 4096 runes 分片，优先在换行符断开。`closeDraft()`（handler.go:347-377）编辑草稿第一条消息为第一段，剩余段用 `sendMessage` 发送。
- **无悬空**：`onDraftClose` 在空内容时调用 `dismissDraft`（handler.go:204-206）删除草稿消息；`onDraftDismiss` 在 `TurnStarted` 时自动关闭前一轮草稿（sink.go:55-58）。
- **证据**：sink.go:250-266 `finalizeTurn()` → onDraftClose。

## 2. Message 不翻倍
**状态：✅ 已实现**

- 草稿先 `sendMessage` 创建，后续编辑用 `editMessageText`（handler.go:315-343 `sendOrEditDraft`）。
- 定稿时编辑已有消息为第一段（handler.go:366-371），而非发送新消息。
- 空内容直接 `dismissDraft` 删除（handler.go:380-405）。

## 3. 审批/Ask 不堵 Sink
**状态：✅ 已实现**

- `event.ApprovalRequest`（sink.go:179-183）：emit 后立即通过 `onApprove` 回调发送 Telegram 消息，不等待用户。
- `event.AskRequest`（sink.go:185-189）：同理，emit 即返回。
- 注释明确标注「§4.9: non-blocking — emit immediately, return」。

## 4. 永久允许后重启仍免问（WORK_DIR 配置可写）
**状态：🔶 部分实现**

- **键盘按钮存在**：`approve.go:52` 有「📌 持久允许」按钮，调用 `ctrl.Approve(id, true, false, true)` 由 reasonix 核心写入 `reasonix.toml`。
- **目录可写检查**：`validatePaths()`（main.go:112-114）检查 `WORK_DIR/reasonix.toml` 父目录可写。
- **缺索引持久化**：`stateStore`（persist.go）定义了存储结构但**从未被实例化或使用**。`SessionManager.index`（session.go:19）纯内存，重启后 chat→path 映射丢失，导致旧 session 文件虽然加密存在于磁盘但无法自动恢复（`ensureController` 会 mint 新路径）。
- **结论**：reasonix.toml 的持久允许由核心写盘，bridge 侧仅检查可写性。但 chat→path 映射未持久化，重启后会从头开始。

## 5. /stop 取消 + dismiss
**状态：✅ 已实现**

- `handler.go:126-128`：`handleCommand` 中 `/stop` → `b.sm.Stop(chatID)` → `ctrl.Cancel()`，发送取消消息。
- `SessionManager.Stop()`（session.go:134-141）：查找活跃 controller 并调用 `Cancel()`。
- **备注**：当前未手动 dismiss 本轮草稿，但下一轮 `TurnStarted` 自动 `onDraftDismiss`（sink.go:55-58）处理前一轮草稿。极微间隙存在，不影响使用。

## 6. /new：Wait 后索引正确；首条消息无索引时能 mint path
**状态：🔶 部分实现**

- **首条消息 mint path**：`ensureController` → `resolvePath()`（session.go:56-67）在 index 为空时调用 `agent.NewSessionPath()` 生成新路径——实现正确。
- **/new 流程不完整**：当前 `/new` 仅调用 `b.sm.Stop(chatID)` + 返回提示（handler.go:164-166），未按设计执行 `Submit("/new") → Wait() → 索引更新`。没有 `/new` 提交给 controller，没有 `Wait`，没有持久化索引更新。
- **索引未持久化**：即使 `/new` 正确执行，chat→path 索引也仅在内存中。

## 7. /model 列表能显示；切换失败不丢会话；成功旧 Close
**状态：🔶 部分实现**

- **列表显示**：当前 `/model`（handler.go:153-162）只显示当前模型名（`cfg.Model` 或 `ctrl.Label()`）。未实现选项 A（Submit "/model" 取 Notice）或选项 B（自读配置列表）。
- **切换**：完全未实现。无 Snapshot → Build(新模型) → Load+Resume → 原子替换 → 旧 Close 流程。
- **失败保留旧会话**：未实现切换逻辑，自然也无不丢会话保护。

## 8. /status API 签名正确
**状态：🔶 部分实现**

- 当前实现（handler.go:141-151）仅展示：`ctrl.Running()`、`ctrl.Turn()`、`ctrl.Label()`。
- 设计要求的缺失字段：
  - `LastUsage() *provider.Usage` ❌
  - `SessionCache() (hit, miss int)` ❌
  - `SessionCost() (float64, string)` ❌
  - `Balance(ctx)` ❌
  - `MultiAgentControl().List(...) []ListedAgent` ❌
  - `SessionPath()` ❌
- Sink 中 `lastUsage`、`sessionCost`、`sessionCache` 字段存在但未暴露给 `/status`。

## 9. `!`/`/compact` 无 TurnStarted 不误关流；`/memory`/`/rewind` 无 Done 不卡死
**状态：🔶 部分实现**

- **无 TurnStarted 保护**：sink.go 各事件处理前都 `if !s.dialogueUI { return }`（line 61、68、76、84、94、119、128、145、148、155、168），无 TurnStarted 时事件被静默丢弃，不会误操作。
- **路由问题**：`/compact`、`/memory`、`/rewind` 不在 `handleCommand` 的 switch 中，走 `default` 分支返回「未知命令」（handler.go:183-185）。它们**未进入 Submit 流程**，该问题与 #10 相同。
- **无 Done 卡死风险**：若通过 Submit 传入只读斜杠但不产生 TurnDone，sink 的 `finalizeTurn()` 永远不会被调用，草稿将保持打开。设计要求「只读斜杠：无 Done，Notice 直接展示」，但当前 sink 没有对「无 Done 只读命令」的特殊处理机制。

## 10. 手打 /memory 走 Submit 生效
**状态：❌ 未实现**

- `/memory` 不在 `handleCommand` switch 中，落入 `default` 返回「未知命令」（handler.go:183-185）。
- 设计明确要求 `/memory`（以及 `/compact`、`/rewind`、`!`、手打 skills/hooks/mcp/branch 等）应走 `Submit` 流程。
- 当前路由逻辑：`handleMessage`（handler.go:90-118）中，所有以 `/` 开头的消息都进 `handleCommand`，而 `handleCommand` 只识别白名单命令，其余返回未知。这导致 `/memory`、`/compact`、`/rewind` 等无法到达 controller。

## 11. 加密 Resume；旧会话可续；损坏换 path
**状态：✅ 已实现**

- **加密层**：`sessioncrypt.go` 实现完整 AES-256-GCM 加密/解密，含魔法头检测、双加密保护、透明读写。
- **迁移**：`persist.go:295-350` `migrateOldSessions()` 将旧 `.jsonl` 转为 `.jsonl.enc`。
- **损坏换 path**：`ensureController`（session.go:95-107）中 `LoadSession` 失败（含损坏）→ 记录日志 → `agent.NewSessionPath` 生成新路径并更新 index。
- **旧会话可续**：`LoadSession` 成功后 `ctrl.Resume(sess, path)`（session.go:109）。
- **缺口**：chat→path 索引不持久化，重启后无法找到旧 session 文件（即使它们加密存在于磁盘）。

## 12. 空闲 Build+Resume
**状态：✅ 已实现**

- `ensureController`（session.go:71-121）在 `Send` 时自动处理：
  - 复用已有活跃 controller（line 73-77）
  - 无可用时 `boot.Build` 创建新 controller
  - `LoadSession` 恢复或全新开始
  - `EnableInteractiveApproval()` 启用交互审批
- 每次 `Send` 调用 `ensureController`，自然支持「空闲后首次消息重建」。

## 13. 无 serve 子进程
**状态：✅ 已实现**

- 全仓库无 `os/exec`、`exec.Command` 或 `reasonix serve` 调用。
- 使用 direct controller embedding (`boot.Build` → `control.Controller`) 替代旧的每聊天 `reasonix serve` 子进程模型。
- `persist.go` 和 `README.md` 中残留的「serve process」注释为旧架构文档未更新。

## 14. 三路径 + 永久允许落点可写；密钥可解析
**状态：✅ 已实现**

- `validatePaths()`（main.go:104-122）检查：
  - `STATE_DIR`（line 106）
  - `REASONIX_SESSION_DIR`（line 107-108）
  - `WORK_DIR`（line 110-111）
  - `WORK_DIR/reasonix.toml` 父目录（line 113-114）
- `checkWritable()`（main.go:124-146）执行实际写测试，目录不存在自动创建。
- 密钥解析：`loadConfig()` 从环境变量加载，从 `/etc/reasonix-telegram.env` 和 `/etc/reasonix-api.env` 读取，TG_BOT_TOKEN 非空校验（line 70-72）。
- **小缺口**：`installSecretLogRedaction()` 定义但从未调用（log_redact.go:35），日志脱敏未启用。

## 15. 未设正数 MaxSteps 时长链不被误杀
**状态：❌ 未实现**

- 全仓库无 `MaxSteps`、`max_steps` 引用。
- `boot.Build` 调用（session.go:80-89）未传递任何 MaxSteps 配置，依赖 reasonix 内部默认值。
- 无显式保护逻辑：既未主动设置安全值，也未检测 reasonix 默认值是否会导致时长链被截断。

## 16. SIGTERM：Wait 后再 Snapshot，重启可续
**状态：✅ 已实现**

- 信号处理：`main.go:183-189` 捕获 SIGTERM/SIGINT → `bridge.Shutdown()`。
- `SessionManager.Shutdown()`（session.go:153-201）严格按设计顺序：
  1. `ctrl.Cancel()`（若 Running，line 170-172）
  2. `ctrl.Wait()`（带 5s 超时保护，line 177-187）
  3. `ctrl.Snapshot()`（line 189-192）
  4. `ctrl.Close()`（line 194-196）
- 注释明确标注对应 §9.2 规范。
- **重启可续**：Snapshot 保存会话内容到磁盘，但如前所述 chat→path 索引不持久化，重启后路径映射丢失（会话文件仍在但 bridge 找不到）。

## 17. environ 无长期明文敏感密钥（允许 *_FILE）
**状态：🔶 部分实现**

- **clear 环境变量**：`loadConfig()`（main.go:91-99）在加载后 clear 四个密钥：TG_BOT_TOKEN、DEEPSEEK_API_KEY、JINA_API_KEY、CF_TOKEN。
- **`*_FILE` 未实现**：main.go:80-82 只有注释，没有实际读取 `DEEPSEEK_API_KEY_FILE`、`JINA_API_KEY_FILE` 等 `*_FILE` 环境变量并从中加载密钥的逻辑。
- **日志脱敏未激活**：`installSecretLogRedaction(log_redact.go:35)` 定义但从未被调用。`cfg.secrets` 仅包含 BotToken（main.go:66）。
- **仅 BotToken 在 secrets 中**：DEEPSEEK_API_KEY、JINA_API_KEY、CF_TOKEN 在 clear 之前若因日志打印会暴露。但 `loadConfig` 在 `godotenv.Load` 后立即 clear，理论上这些值在被日志记录前已清除——但仍存在窗口期。

## 18. 断连可降级 sendMessage
**状态：🔶 部分实现**

- **底层重试**：`TelegramClient.doWithRetry`（telegram_client.go:273-303）对网络错误和 429/5xx 进行最多 3 次指数退避重试。
- **草稿编辑降级未实现**：`sendOrEditDraft`（handler.go:315-343）在 `editMessageText` 失败时仅记录日志并忽略错误（line 338-341），没有降级到 `sendMessage` 的 fallback 路径。
- `sendMessage`（handler.go:411-416）是简单封装，无重试逻辑（调用 `client.Send` 后仅 log 错误）。
- README 提到「草稿 API 不可用时回退为 sendMessage + editMessageText」但代码未体现此降级策略。

## 19. 定时失败有用户可见反馈；投递失败有限重试
**状态：🔶 部分实现**

- **定时任务**：`cron.go` 和 `cron_serve.go` 均为 stub（仅打印"will be implemented in step 8"），定时/定时任务功能完全未实现。
- **无定时失败反馈**：因定时任务未实现。
- **投递重试**：底层 `doWithRetry` 实现了 3 次有限重试（telegram_client.go:273-303），覆盖 429/5xx/网络错误。
- **用户可见重试反馈**：sink.go:168-177 处理 `event.Retrying`，在草稿中显示「正在重试 (N/M)…」消息——这由 reasonix 核心的 Retrying 事件驱动，桥侧渲染正确。
