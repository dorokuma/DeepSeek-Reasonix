# Audit: Cache Hit Rate Display & Event Passing

## Scope
Review of cache-hit rate calculation, event emission, and frontend display across agent, CLI, serve (REST + Web UI), and metrics.

---

## 1. Agent: `internal/agent/agent.go`

### 1a. Session accumulators (lines 185–195, 857–861)

```go
sessCacheHit  atomic.Int64   // cumulative cache-hit prompt tokens
sessCacheMiss atomic.Int64   // cumulative cache-miss prompt tokens
```

**Accumulation** (line 860–861 in the streaming loop):
```go
case provider.ChunkUsage:
    a.sessCacheHit.Add(int64(chunk.Usage.CacheHitTokens))
    a.sessCacheMiss.Add(int64(chunk.Usage.CacheMissTokens))
```

✅ **Correct** — per-turn CacheHitTokens/CacheMissTokens are added to the session cumulative counters after each API call. Every streaming provider (OpenAI, Anthropic) emits exactly **one** `ChunkUsage` per stream, so there is no double-counting risk.

### 1b. Usage event emission (lines 612–625)

```go
SessionHit:   int(a.sessCacheHit.Load()),
SessionMiss:  int(a.sessCacheMiss.Load()),
```

The event carries:
- `Usage` → the **per-turn** `*provider.Usage` (last chunk's usage)
- `SessionHit`/`SessionMiss` → the **session cumulative** values (not per-turn!)

✅ **Correct** — the wire protocol separates per-turn (`cacheHitTokens`) from session-cumulative (`sessionCacheHitTokens`).

### 1c. Sub-agent merge (`AddSessionUsage`, line 362–366)

```go
func (a *Agent) AddSessionUsage(hit, miss, prompt, total int64, ...) {
    a.sessCacheHit.Add(hit)
    a.sessCacheMiss.Add(miss)
    ...
}
```

Called by `task.go`'s `mergeSubUsage()` (line 371–386) after sub-agent finishes:
```go
hit := sub.sessCacheHit.Load()
miss := sub.sessCacheMiss.Load()
parentAgent.AddSessionUsage(hit, miss, prompt, total, cost, currency)
```

✅ **Correct** — sub-agent's session cumulative values are merged into parent's accumulators. The sub-agent's own Usage events **do not** reach the frontend (they are dropped by `subSinkFor`, line 459–470 in `task.go`). The parent's **next** Usage event will include the merged values through `SessionHit`/`SessionMiss`.

### 1d. Coordinator planner merge (coordinator.go lines 173–196)

```go
deltaHit := currHit - c.plannerPrevHit
deltaMiss := currMiss - c.plannerPrevMiss
c.executor.AddSessionUsage(deltaHit, deltaMiss, ...)
```

✅ **Correct** — uses delta (not full cumulative) so the planner's tokens are only merged once per plan cycle.

## 2. TUI: `internal/cli/chat_tui.go`

### 2a. `cacheTag()` (lines 2294–2316)

**"now" (single-turn) rate:**
```go
u := m.ctrl.LastUsage()  // *provider.Usage — the most recent API call's usage
d := u.CacheHitTokens + u.CacheMissTokens
if d == 0 { d = u.PromptTokens }
now = cacheRateLabel(..., u.CacheHitTokens, d)
```

- Reads from `LastUsage()` → **last turn's per-hit/miss** ✅
- Fallback to `PromptTokens` when `CacheHitTokens + CacheMissTokens == 0` → shows `0%` when no cache info, **not a bug** but mildly misleading label (it's `hit/denom` with `denom=PromptTokens` which is not a hit-rate in the strict sense). However, this only triggers when cache tokens are unreported, and `CacheHitTokens` is 0, so `0/anything = 0%` is correct.

**"avg" (session aggregate) rate:**
```go
hit, miss := m.ctrl.SessionCache()  // session-cumulative
avg = cacheRateLabel(..., hit, hit+miss)
```

✅ **Correct** — uses `SessionCache()` which returns `sessHit/(sessHit+sessMiss)`.

### 2b. `cacheRateLabel()` (lines 2282–2287)

```go
func cacheRateLabel(format string, hit, denom int) string {
    if denom <= 0 { return "" }
    return fmt.Sprintf(format, fmt.Sprintf("%.2f%%", float64(hit)*100/float64(denom)))
}
```

✅ **Division-by-zero guard** (`denom <= 0`). Uses `%.2f%%` for 2-decimal precision.

## 4. Wire Protocol: `internal/serve/wire.go`

### 4a. `wireUsage` struct (lines 72–93)

| Field | Source | Kind |
|-------|--------|------|
| `cacheHitTokens` | `e.Usage.CacheHitTokens` | **per-turn** |
| `cacheMissTokens` | `e.Usage.CacheMissTokens` | **per-turn** |
| `sessionCacheHitTokens` | `e.SessionHit` | **session cumulative** |
| `sessionCacheMissTokens` | `e.SessionMiss` | **session cumulative** |

✅ **Correct mapping** (lines 171–177).

### 4b. Unused fields in frontend

⚠️ **Observation**: The `sessionCacheHitTokens`/`sessionCacheMissTokens` fields in `wireUsage` are **not consumed** by the frontend JS (`index.html`). They are transmitted but never read. The frontend instead:
- Accumulates per-turn `cacheHitTokens` from Usage events into `cumulativeCacheHit`
- Falls back to `/status` API's `cacheHit`/`cacheMiss` (session-cumulative)

This is **not a bug** — just dead data in the wire format. The frontend's approach works correctly because per-turn accumulation should equal session-cumulative values after all events have been received.

## 5. Frontend: `internal/serve/index.html`

### 5a. Per-turn accumulation in SSE handler (lines 864–865)

```js
cumulativeCacheHit+=e.usage.cacheHitTokens||0;
cumulativeCacheMiss+=e.usage.cacheMissTokens||0;
```

✅ **Correct** — accumulates per-turn `cacheHitTokens` from each Usage event. Since the frontend only sees parent-agent events (sub-agent events are filtered by `subSinkFor`), and the parent's per-turn cache tokens match the session addition, this should equal the session cumulative value.

### 5b. Metric strip single-turn rate (line 870)

```js
Math.round(e.usage.cacheHitTokens/Math.max(1,e.usage.cacheHitTokens+e.usage.cacheMissTokens)*100)
```

✅ **Division-by-zero protection** via `Math.max(1, ...)`. Shows the single-turn hit-rate percentage.

### 5c. Sidebar cache rate via status API (lines 893–894)

```js
const cacheTotal=(s.cacheHit||0)+(s.cacheMiss||0);
$('#sm-cache').textContent=cacheTotal>0?Math.round((s.cacheHit||0)/cacheTotal*100)+'%':'—';
```

✅ **Correct** — uses `/status` session-cumulative values. Shows `—` when no cache data.

### 5d. Stats modal (lines 1005–1006)

```js
const hit=cumulativeCacheHit||s.cacheHit||0, miss=cumulativeCacheMiss||s.cacheMiss||0;
const rate=hit+miss>0?Math.round(hit/(hit+miss)*100)+'%':'0%';
```

✅ **Correct** — prefers frontend-cumulative over status API, with zero protection.

⚠️ **Subtlety**: `cumulativeCacheHit` is `0` on page load. `0 || s.cacheHit || 0` evaluates to `s.cacheHit || 0` because JS treats `0` as falsy. This is **acceptable** because:
- If no cache events occurred, `s.cacheHit` is also `0` → shows `0%` (or `'0%'` via the ternary)
- If cache events occurred but page was loaded mid-session, `s.cacheHit` provides the correct fallback

However, if `cumulativeCacheHit` is legitimately `0` and `s.cacheHit` is non-zero (e.g., history session with cache but no new usage), the stats modal shows the status API value correctly. No issue.

## 7. Potential Issues & Concerns

### P2: `sessionCacheHitTokens`/`sessionCacheMissTokens` never consumed by frontend
- **File**: `internal/serve/wire.go` lines 82–83, `internal/serve/index.html`
- **Detail**: These fields are emitted in every Usage event over SSE but the frontend JS never references them. They account for ~15–20% of the `wireUsage` JSON payload's size.
- **Recommendation**: Remove from `wireUsage` or document as "reserved for future use."

### P2: Transient divergence between frontend accumulation and backend session values
- **Files**: `internal/serve/index.html` lines 864–865, `internal/agent/agent.go` line 860
- **Detail**: When a coordinator is used, the planner agent emits a Usage event (with per-turn `cacheHitTokens`) that reaches the frontend. The `cumulativeCacheHit` is incremented by the planner's per-turn value. Then, when the executor emits its Usage event, `cumulativeCacheHit` is incremented again by the executor's per-turn value. Between these two events, `cumulativeCacheHit` under-reports relative to the status API's `cacheHit` (which includes both via the delta merge). This inconsistency lasts only until the executor's event arrives (seconds at most).
- **Impact**: Very low — user might briefly see a slightly different rate in the stats modal vs. the sidebar.
- **Recommendation**: The frontend could also update `cumulativeCacheHit` from the status API during `fetchStatus()` calls, which already refreshes the sidebar correctly.

### P3: Integer division truncation in test vs. rounding in production
- **File**: `internal/agent/cachehit_e2e_test.go` line 349 vs chat_tui.go line 2286 / index.html line 1006
- **Detail**: The end-to-end test uses `100 * hit / (hit + miss)` (integer division, truncates). Production TUI uses `float64(hit)*100/float64(denom)` with `%.2f` (rounds to 2 decimals). Frontend uses `Math.round(hit/(hit+miss)*100)` (rounds to integer). These can differ by up to ~0.5 percentage points in edge cases.
- **Impact**: None — the test only checks that `agg > 0 && agg <= 100`. Production rounding is consistent per channel.
- **Recommendation**: No action needed.

### P4: Frontend `||` operator with zero values
- **File**: `internal/serve/index.html` lines 1005
- **Detail**: `cumulativeCacheHit||s.cacheHit||0` — JS `||` treats `0` as falsy. If `cumulativeCacheHit` is `0` (no usage events received yet) and `s.cacheHit` is also `0` (API returns 0), the expression evaluates to `0`, which is correct. But if `cumulativeCacheHit` is `0` and `s.cacheHit` is `100`, it evaluates to `100`, which is also correct. The only edge case would be if `cumulativeCacheHit` were legitimately `0` from a session with absolutely no cache tokens (all turns had zero cache tokens) — in that case, `0 || 0 || 0 = 0`, and the rate would show `0%` which is correct.
- **Impact**: None.
- **Recommendation**: Use `??` (nullish coalescing) instead of `||` to avoid treating legitimate `0` values as "absence of data." E.g., `(cumulativeCacheHit ?? s.cacheHit ?? 0)`. But this is a style/minor-correctness issue.
